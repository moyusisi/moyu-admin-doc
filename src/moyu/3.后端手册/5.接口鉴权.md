::: tip
鉴权就是识别出来"你能做什么"，是认证鉴权中的鉴权部分，可用`authZ(Authorization)`表示。
权限管理就是控制用户可以访问那些资源，包括前端的菜单、页面、按钮以及后端的数据接口。
:::

## 权限模型
系统在 `RBAC（Role-Based Access Control）` 权限模型的基础上引入了`角色组(Group)`的概念，实现`用户-角色-权限`或`用户-岗位-角色-权限`的灵活访问控制，涵盖后端接口鉴权与前端菜单按钮权限。

### 核心概念

| 概念               | 说明                                             |
| ------------------ | ------------------------------------------------ |
| **用户(User)**     | 系统操作者，通过角色/角色组间接获得权限          |
| **角色(Role)**     | 一组资源权限的集合                               |
| **资源(Resource)** | 系统功能入口，如目录、菜单、按钮等，关联权限标识 |
| **权限标识(Perm)** | 功能权限的唯一标识，如`sys:user:add`             |
| **角色组(Group)**  | 一组角色的集合，本系统中用**岗位**表示           |

### 使用方式
权限管理时有以**角色**为核心和以**岗位**为核心两种不同的使用形式，不同形式的特点如下

| 权限管理核心    | 权限链条              | 适用场景                               |
| --------------- | --------------------- | -------------------------------------- |
| **角色(Role)**  | `用户-角色-权限`      | 适用于组织机构复杂、用户较多的复杂场景 |
| **岗位(Group)** | `用户-岗位-角色-权限` | 适用于组织机构简单、用户较少的简单场景 |


* 以角色为核心进行权限管理时，支持**一人多角色**，用户可同时具有多个角色的权限，无法进行角色切换。
* 以岗位为核心进行权限管理时，支持**一人多岗**、**一岗多角**，用户可以进行**岗位切换**，岗位切换后对应的权限也会变化。
* 两种形式同时使用时，将`用户-角色`关系直接当作**系统默认岗位**处理，可参与岗位切换。

## 前端资源权限

### 菜单权限
菜单页面的展示由前端的**动态路由**功能自动从后端接口`/userMenu`获取并渲染

### 按钮权限

用户登录后通过`/userInfo`接口获取并保存在本地的用户信息`USER_INFO`中已经包含用户的权限相关字段`roles`和`perms`

**权限存储：**
```js {5-15} title="USER_INFO"
{
    "account": "superAdmin",
    "orgCode": "11000000",
    "name": "超管小李",
    "roles": [
        "r_superAdmin",
        "r_default"
    ],
    "perms": [
        "sys:user:page",
        "sys:user:detail",
        "sys:org:page",
        "sys:org:detail",
        ...
    ],
    "groupCode": "g_default",
    ...
}
```
**使用方式：**
前端页面中若要控制按钮权限可使用`@/utils/permission`中的函数`hasPerm`、`hasAnyPerm`、`hasRole`等

```js
import { hasPerm } from "@/utils/permission"
// js脚本中：
if (hasPerm(['sys:user:add'])) ...
// vue模板中：
<a-button v-if="hasPerm('sys:user:add') || hasRole(['admin'])">操作</a-button>
```

## 后端接口鉴权
系统使用了[Sa-Token](https://sa-token.cc/doc.html#/use/jur-auth)进行权限认证，支持注解鉴权，同时本系统内置了一个全局过滤器，可以配置需要鉴权的URL。

### 全局配置鉴权
在全局过滤器中，会对一些URL进行登录检查(checkLogin)，可以通过配置进行控制

```yml
# 自定义配置
custom:
  # 自定义认证鉴权配置
  auth:
    # 是否启用鉴权，为false时等同于全加白
    enabled: true
    # 需要进行认证的列表，默认["/api/**"]
    auth-list:
      - /api/**
    # 白名单列表，默认为空列表[]
    white-list:
      # 认证授权类
      - /api/auth/login/**
      # 回调、远程调用类
      - /callback/**
```
### 注解鉴权
本系统系统支持`Sa-Token`的[AOP注解鉴权](https://sa-token.cc/doc.html#/plugin/aop-at)，常用的注解如下表，更多可参考[Sa-Token官方注解](https://sa-token.cc/doc.html#/use/at-check)

* `@SaCheckLogin`: 登录校验 —— 只有登录之后才能进入该方法。
* `@SaCheckRole("admin")`: 角色校验 —— 必须具有指定角色标识才能进入该方法。
* `@SaCheckPermission("sys:user:page")`: 权限校验 —— 必须具有指定权限才能进入该方法。

## 权限模型关系表

无论是功能权限还是数据权限，均通过关系表进行关联，其表结构如下：

```sql
-- 用户角色权限关系表
CREATE TABLE `sys_relation`
(
    `id`            BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键id',
    `object_id`     VARCHAR(64)   DEFAULT NULL COMMENT '对象ID',
    `target_id`     VARCHAR(64)   DEFAULT NULL COMMENT '目标ID',
    `relation_type` TINYINT       DEFAULT NULL COMMENT '关系类型(字典 1:user_has_role,2:role_has_perm,3:user_has_group,4:group_has_role)',
    `ext_json`      TEXT          DEFAULT NULL COMMENT '扩展信息',
    ...
    PRIMARY KEY (`id`)
) CHARACTER SET = utf8mb4 COLLATE = utf8mb4_general_ci COMMENT = '用户角色权限关系表';
```
